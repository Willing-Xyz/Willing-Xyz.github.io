实现自定义View的滚动

View滚动原理

首先，View滚动本质上是根据View的scrollTo和scrollBy方法来进行滚动的。调用这两个方法时，改变的只是mScrollx和mScrolly的值，而x和y的并没有改变。所以如果在View实现滚动，在onDraw方法中，要考虑mScrollx和mScrolly的值。

scrollTo和scrollBy滚动的是View的内容，View的位置并不变化。也就是说，在onDraw方法中，根据mScrollx/y的值，来绘制不同的内容。

scrollTo和scrollBy参数的意义：
当调用这两个方法时，mScrollx和mScrolly的值会根据参数变化。
它们的值等于scrollTo的参数值，或者等于它们的值加上scrollBy的参数值。
它们的值可通过getScrollX和getScrollY返回。
当参数值为正时，表示View的内容向上或向左滚动，当为负时，表示View内容向下或向右滚动。

Scroller类

Scroller类主要提供参数给scrollTo和scrollBy
例如， void startScroll (int startX, int startY, int dx, int dy, int duration)方法，它指定了开始坐标，和滚动的距离和完成滚动所需的时间（它并不完成View的滚动，它只是计算在duration期间，x和y值的变化），然后通过调用getCurrX和getCurrY方法获取滚动的坐标，通过scrollTo和scrollBy方法进行实际的滚动。

另外根据不同的Interpolator可以实现不同的效果，例如使用BounceInterpolator可以实现弹跳的效果。

OverScroller类

OverScroller相比Scroller添加了springBack方法。该方法能够把偏移值限制在某个范围内。

```
Collection Map

简介

Colleciton的元素类型必须实现equals方法和hashCode方法。

所有通用的Collection实现类应该提供两个“标准”构造方法：

  • 一个是void参数的构造方法，用于创建空Collection；
  • 另一个是带有Collection类型单参数的构造方法，用于创建一个具有与其参数相同元素新的Collection。

Collection接口中的“破坏性”方法，是指可修改其所操作的collection的那些方法，如果此collection不支持该操作，则抛出UnsupportedOperationException异常。这样的方法标为“可选”。

并发容器

并发容器相对同步容器有更高的性能，而且由于它增加了一些常用的符合操作，所以也会降低使用的风险。

Iterator

所有集合类都实现了Iterable接口，该接口中包含方法Iterator<T> iterator()，返回Iterator，表示可用它来遍历数据。

对于非线程安全的集合类和使用Collections.synchronizedXXX创建出来的，以及Vector,Stack,Hashtable（不包括EnumSet和EnumMap），如果在迭代期间计数器被修改，那么hasNext或next将抛出ConcurrentModificationException异常。然而，这种检查是在没有同步的情况下进行的，因此可能会看到失效的计数值，而迭代器可能并没有意识到已经发生了修改。

另外，其他的操作，如toString，hashCode，equals等方法都会进行迭代，也就是有抛出ConcurrentModificationnException异常的可能。

为了避免该异常，可以考虑遍历它的副本。

并发容器和EnumSet和EnumMap的迭代是弱一致性的。它们不会抛出ConcurrentModificationException，因此不需要再迭代过程中加锁。
弱一致性的迭代器可以容忍并发的修改，当创建迭代器时会遍历已有的元素，并可以（但不保证）在迭代器被构造后将修改反映给容器。

List接口

List接口按照添加元素的顺序排列。

List接口返回的迭代器为ListIterator，它包含向前和向后的迭代。

RandomAccess接口

该接口为一个标记接口，内部没有方法。

Vector, ArrayList，Stack

Vector和ArrayList都支持快速访问，内部使用数组实现，区别在于Vector是线程安全的，而ArrayList不是。

Stack表示堆栈，继承自Vector。
在设计上，Stack不应该扩展Vector，而是使用组合来实现。

另外，Vector和Stack是遗留类，应该使用其他类来代替。

LinkedList

LinkedList内部使用双向链表实现，支持在头部和尾部插入、删除、get元素。

另外，它实现了Deque接口，支持双端队列操作。
在使用该接口时，不应该包含null值。
因为poll操作表示移除并返回队列的值，如果队列为空，返回null。
所以如果，包含了null值，则不能够判断返回null是元素的值还是表示队列为空。

CopyOnWriteArrayList

当调用iterator返回迭代器时，迭代器保存一个指向底层数组的引用，当其他线程修改容器时，容器会复制底层数组并修改相应的数据，这就迭代器中的元素就和它创建时完全相同。

仅当迭代操作远远多于修改操作时，才应该使用该类。例如用来保存监听器列表。

Queue接口

该接口提供两种形式的操作：

  • 一种是抛出异常
  • 之中是返回特殊值表示结果

在使用该接口时，不应该包含null值。
因为poll操作表示移除并返回队列的值，如果队列为空，返回null。
所以如果，包含了null值，则不能够判断返回null是元素的值还是表示队列为空。

Deque

支持在队列的两端插入和删除元素，同样提供两种形式的操作，形式为方法名加上First或Last。
其次，它包含了用于堆栈的方法，所以应该有限使用此接口而不是Stack类。

ArrayQueue

不允许包含null元素。

内部使用数组实现，由于实现了Deque接口，所以支持队列和堆栈的操作。

PriorityQueue

不允许包含null元素。

该类包含的类型应该实现Comparable接口或者提供Comparator的实现。

BlockingQueue

增加了阻塞方法和超时方法。

BlockingDeque

支持在队列的两端插入和删除元素，同样提供两种形式的操作，形式为方法名加上First或Last。

双端队列可以用来实现工作密取（Work Stealing）。
例如，每个消费者有自己的双端队列，如果一个消费者完成了自己队列中的工作，那么它可以从其他消费者队列的尾部秘密的获取工作。
这样一个从头部，一个从尾部获取工作，导致了更少的数据竞争，提升了性能。

ArrayBlockingQueue

由数组实现的阻塞队列。
它是有界的。一旦初始化后，就不能再增加容量。

LinkedBlockingQueue，LinkedBlockingDeque

由列表实现的阻塞队列。
它是有界的。一旦初始化后，就不能再增加容量。
如果没有指定容量，则默认为Integer.MAX_VALUE。

PriorityBlockingQueue

该队列是无界的。

DelayedQueue

此队列不允许使用 null 元素。
Delayed 元素的一个无界阻塞队列，只有在延迟期满时才能从中提取元素。该队列的头部 是延迟期满后保存时间最长的 Delayed 元素。如果延迟都还没有期满，则队列没有头部，并且 poll 将返回 null。当一个元素的 getDelay(TimeUnit.NANOSECONDS) 方法返回一个小于等于 0 的值时，将发生到期。即使无法使用 take 或 poll 移除未到期的元素，也不会将这些元素作为正常元素对待。例如，size 方法同时返回到期和未到期元素的计数。

SynchronousQueue

每一个put操作必须等待另一个线程的poll操作，反之亦然。

Set接口

Set中不包含重复元素。

HashSet，LinkedHashSet

HashSet允许包含至多一个null元素。

LinkedHashSet允许包含至多一个null元素。

Java并发编程实践笔记-线程安全

核心思想

要编写线程安全的代码，其核心在于要对状态访问操作进行管理，特别是对共享的和可变的状态的访问。
其次，对于只读对象，也要保证它的可见性。

对象的状态：包含了任何可能影响其外部可见行为的数据。

共享：变量可以由多个线程同时访问。

可变：变量的值在其生命周期内可以发生变化。

竞态条件

竞态条件：当某个计算的正确性取决于多个线程的交替执行时序时，就会发生竞态条件。

常见的竞态条件

  • 先检查后执行：通过一个可能失效的观测结果来决定下一步的动作（在观测结果为真和执行下一步动作之间，观测结果可能会失效）。
  • 读取-修改-写入：基于对象之前的状态来定义对象状态的转换（例如i++可分为三步。但i=1，就没有基于对象之前的状态）。
  • 不是彼此独立的多个变量：如两个变量应该同时更新，要么都不更新。

解决方法： 以原子方式执行该组操作。

多线程中可能存在的问题

  • 原子性：必须以原子方式执行的操作可能被分开执行
  • 可见性：一个线程写入的值，不能立即被另一个线程读取
  • 重排序：编译器和处理器为了优化性能，可能把没有依赖关系的操作重排序

解决多线程问题的方法

  • 锁机制：锁解决了原子性问题，可见性问题，和重排序问题。
  • volatile关键字：volatile解决了可见性问题，重排序问题，另外保证了long和double变量可以原子性的执行单步读或写操作。

final关键字

对于包含final字段的对象，只要对象是正确构造的（被构造对象的引用在构造函数中没有“逸出”），那么不需要使用同步（指lock和volatile的使用），就可以保证任意线程都能看到这个final字段在构造函数中被初始化之后的值（包括final字段内部的状态）。
如果final字段引用的是可变对象，那么在访问可变对象的内部状态时仍然需要同步。

发布与逸出

  • 发布：使对象能够在当前作用域之外的代码中使用，并且通过该对象能够访问的所有对象也会被发布。
  • 逸出：某个不应该发布的对象被发布时，这种情况被称为逸出。
  • 未被正确发布：发布时没有使用同步来确保对象对其他线程可见。

逸出-不正确构造：

如果this引用在构造过程中逸出，那么这种对象就被认为是不正确构造。

  • 在构造函数中把内部类实例发布（因为内部类实例中包含了对外部类实例的引用），如注册监听器，这使得在构造函数还未完成时，使尚未完全构造的对象发布出去。
  • 在构造函数中启动一个线程（如果该线程是内部类，或者传递this到线程的构造函数）。
  • 在构造函数中调用给一个可override的实例方法（既不是private方法也不是final方法），因为如果子类override该方法，而把未构造完的this置于一个未知的环境（因为子类可以任意override）。

如果想要在构造函数中注册一个事件监听器或启动线程，那么可以使用一个私有的构造函数和一个公共的工厂方法。

  1. 创建一个私有变量，来保存监听器或线程。
  2. 创建一个私有构造函数，在其中创建监听器或线程实例。
  3. 创建一个公有newInstance/init方法，在该方法中启动线程或注册监听器。

安全的发布

安全的发布：保证对其他线程的可见性（否则，当某线程初始化了某对象后，另外的线程可能看到不一致的状态）

  • 在静态初始化器中初始化一个对象引用（即static变量在声明时或static块中初始化）
  • 将对象的引用保存到volatile类型的域或者AtomicReferance对象（那么AtomicReferance变量本身也需要被安全发布）中。
  • 将对象的引用保存到某个正确构造对象的final类型域中。
  • 将对象的引用保存到一个由锁保护的域中。

不可变对象：

  • 对象创建以后其状态就不能修改
  • 对象的所有域都是final类型（也可以不是）
  • 对象是正确创建的（在对象创建期间，this引用没有逸出）

事实不可变对象：对象技术上是可改变的，但发布之后不会再改变。

  • 不可变对象必须通过安全方式发布，之后的访问不需要同步。
  • final对象可以以任一形式发布，如果引用的对象可变，仍然需要同步。
  • 事实不可变对象：必须通过安全方式发布，之后的访问不需要同步。
  • 可变对象：只能确保发布时的可见性，之后对它的访问依然需要同步（或者根据条件使用volatile）。

在现有的安全类中添加功能

  • 扩展X类（使用X类的锁，在访问/修改X类的状态时）
  • 客户端加锁（包含X类的一个实例，并且使用X类内部使用的锁来同步）
    以上两种方法把同步策略实现分布在多个源代码文件中，如果底层的类改变了同步策略并选择了使用不同的锁来保护它的状态变量，那么子类的安全性会被破坏。
  • 组合（实现一个类，把原有的操作委托给X类，并使用自己的锁来同步）
    推荐使用该方法。
    这种方法不用关心底层类的加锁机制，它在外部又加上一个锁。

Java并发编程实践笔记-概念

并发与并行

  • 并发（concurrent）：多个线程在同一个CPU上交替执行。
  • 并行（parallel）：多个线程在各自的CPU上执行。

多线程的优势

  • 发挥多处理器的强大能力（即使在单处理器上也能获得更高的吞吐率）。
  • 建模的简单性（为每个任务分配一个线程，并在需要时进行交互）。
  • 异步事件的简单处理（如服务器对每个请求创建一个线程（一般有最大线程数量的限制））。
  • 响应更灵敏的用户界面

多线程带来的风险

  • 安全性：产生正确的结果
  • 活跃性：正确的结果最终会发生（如死锁）
  • 性能：正确的结果尽快发生

最低安全性

当线程在没有同步的情况下读取变量时，可能是一个失效值，但至少这个值是由之前某个线程设置的值，而不是一个随机值。

最低安全性适用于大多数变量，但存在一个例外：非volatile类型的64位数值变量（double和long）。使用volatile可以保证它们的最低安全性。

ThreadLocal

ThreadLocal的作用

ThreadLocal是为了隔离多个线程的数据共享。每个线程拥有自己的对象，不会和其他线程发生数据竞争。
另外，它们之间也不应该发生数据竞争，因为既然使用了ThreadLocal，说明每个线程都有自己的数据，并且和其他的线程不同。
所以，ThreadLocal并不是为了解决数据竞争的问题，解决数据竞争是指用同步、锁等来协调多个线程对同一个数据的访问，而使用ThreadLocal时，每个线程存储的是不同的数据。
它只是能够在不同的线程下，通过同一个ThreadLocal访问到各自的数据。

ThreadLocal的方法

ThreadLocal的实现原理

首先，每个线程中有一个ThreadLocalMap类型的成员变量threadLocals。
threadLocals的key为ThreadLocal，值为该线程对应的值。
所以，每个线程的值保存在线程内部，而不是ThreadLocal中。

为什么会这么设计？为什么Map不保存在ThreadLocal中，而是保存在Thread中？
可能是因为如果保存在ThreadLocal中，当线程结束后，其对应的值不能被垃圾回收，而保存的Thread中，可以保证在Thread结束时，对应的值被垃圾回收。

ThreadLocal的作用场景

ThreadLocal经常作为private static变量。
当它所在的类需要被多个线程使用，而每个线程使用各自的数据时，应该使用ThreadLocal。

ThreadLocal可能引发的内存泄漏

首先，可以看到，Entry扩展WeakReference，其中Key为WeakReference，而Value为强引用。
![][/images/threadlocal-leak.png]
所以，当ThreadLocal没有外部强引用时，当系统gc时，ThreadLocal会被回收，key为null，那么对应的value就不能被访问，也不会被释放（除非Thread被销毁）。
但是，ThreadLocalMap的实现中，有一些避免内存泄漏的方法。
在调用ThreadLocal的get或set方法时，会根据hashcode寻找key的位置，在寻找的过程中，如果发现key为null，而value不为null，会释放value的强引用。
当然，在不能保证不存在内存泄漏，所以在不用时，调用ThreadLocal的remove方法来防止内存泄漏。

Handler

基本要素

  • Message：消息的表示
  • MessageQueue：消息队列
  • Looper：消息循环，用于从消息队列中取出消息
  • Handler：消息处理

Handler的使用

  1. 初始化
    为了使用Handler，必须初始化。
  • 在主线程中，已经预先初始化了，所以可以直接使用Handler.
  • 而在自己创建的线程中，需要首先调用Looper.prepare();方法用来初始化MessageQueue和实例化Looper并设置Looper作为ThreadLocal的值，然后调用Looper.loop()；方法进行消息的循环和处理。
  • 或者在构造时使用已经初始化的Looper，这样的话，消息的处理依然是在Looper初始化时所在的线程。

  1. 扩展Handler或者实现Handler.Callback接口作为Handler的构造参数，并实现handleMessage方法。实例化Handler子类时，应该在prepare和loop之间进行。

  2. 获得Handler对象实例，调用方法发送消息到消息队列。消息可以是Runnable或Message类型，分别是使用post和send方法。

HandlerThread的使用

因为HandlerThread的run方法中，已经调用Looper的prepare和loop方法，这个线程已经在等待消息了。
Handler的构造函数中，使用了上面线程的Looper，因此，发送给Handler的消息会放到上面线程的Queue中，且在上面线程中执行。

post和sendMessage

post发送的Runnable最终被封装为Message，其中Runnable作为Message的callback域的值。
然后当Looper的消息循环收到消息时，根据Message的target域中的值，会调用对应Handler的dispatchMessage方法。
从方法中可以看到，如果Message的callback不为空（也就是使用post发送），那么直接调用Runnable的run方法，也就是说，在Looper所在的线程中同步处理。
否则，则在handleMessage方法中处理。默认的handleMessage方法为空。

Handler可能引起的内存泄漏和解决方案

首先，Looper的prepare方法会设置Looper到ThreadLocal中，而ThreadLocal变量作为Looper的static变量，又因为Message的target字段中包含Handler的引用，而Message被放入消息队列，所以导致Handler的生命周期比Activity长。

如果Handler在Activity作为内部类实现，那么就会保存一个Activity的引用。或者消息队列中包含尚未处理的Message，当Activity销毁后，导致Activity不能被垃圾回收。
解决办法：

  1. Handler作为静态类或外部类实现
  2. Activity作为参数传递给Handler的构造器
  3. Handler内部使用软引用或弱引用来保存Activity。
  4. 在handleMessage或run方法中使用Activity时，应该判断它是否为空。
  5. 另外，在Handler的post参数Runnable，也不应该使用内部类。

另外，当Activity销毁时，还在消息队列中排队的任务应该被取消掉，否则Activity还是不能被垃圾回收。可以通过调用Handler中的remove..方法或者mHandler.removeCallbacksAndMessages(null); 删除所有的Runnable和Message。

View/ViewGroup的绘制流程与它们的自定义

View绘制流程

  1. measure：用于计算View的长度和宽度。其中会调用onMeasure方法，子类通过重写该方法，可以实现自己的计算View大小的方式，最后必须调用setMeasureDimension保存计算结果。

  2. layout：用于设置View在屏幕上显示的位置。

  3. draw：将View显示在屏幕上

    1. 绘制背景
    2. 如果要视图显示渐变框，这里会做一些准备工作
    3. 绘制视图本身，即调用onDraw()函数。
    4. 如果是ViewGroup，则绘制子视图，即dispatchDraw()函数。
    5. 如果需要（应用程序调用了setVerticalFadingEdge或者setHorizontalFadingEdge），开始绘制渐变框；
    6. 绘制滚动条；

requestLayout和invalidate的区别

invalidate和requestLayout方法的调用都会向上传递，导致整棵View树被重绘或者重新布局。

自定义View/ViewGroup

  1. 自定义View属性（可选）
  2. 在自定义View类的构造方法中获取自定义的属性值（可选）
  3. 实现onMeasure方法
  4. 实现onLayout方法（自定义ViewGroup需要实现此方法，自定义View不需要）
  5. 实现onDraw方法（自定义View需要，自定义ViewGroup不需要）

1. 在res/values目录下创建xml文件，并定义属性值

既可以使用attr定义属性，也可以包含在declare-styleable中。
区别在于，包含在declare-styleable中，会自动生成一个名为Customize的int数组，其中包含内部attr声明的值。
这样就可以直接作为Context.obtainStyledAttributes的第二个参数，否则要自己根据attr构建数组。
在该xml文件中，设置属性和声明属性 , 也可以声明系统属性，如 , 此时不需要声明format，然后在类中这么获取：ta.getString(R.styleable.test_android_text);
布局文件中直接android:text=”@string/hello_world”即可。

支持的类型

2. 在构造函数中获取属性的值

如果在Code中实例化一个View会调用第一个构造函数，如果在xml中定义会调用第二个构造函数，而第三个函数系统是不调用的，要由其它的构造函数显示调用。

obtainStyledAttributes函数第三个参数，也就是构造函数的第三个参数，它的值为一个属性值，如

protected void onMeasure(int widthMeasureSpec, int heightMeasureSpec)
{
int widthMode = MeasureSpec.getMode(widthMeasureSpec);
int widthSize = MeasureSpec.getSize(widthMeasureSpec);
int heightMode = MeasureSpec.getMode(heightMeasureSpec);
int heightSize = MeasureSpec.getSize(heightMeasureSpec);
// ………..
// (自定义ViewGroup时调用)measureChildren(widthMeasureSpec, heightMeasureSpec); 或者measureChild(child, widthMeasureSpec, heightMeasureSpec);
setMeasuredDimension(width, height);
}
```
其中Mode有三个值：
EXACTLY：表示View的大小应该为上面获取的Size的大小
AT_MOST：表示View的大小最大为Size的大小
UNSPECIFIED：表示View的大小可以随意指定

最后，必须调用setMeasuredDimension方法设置测量的值。
该方法被调用后，Mesaured width/height的值就可以被获取了。

此外应该确保measure height and width 至少是getSuggestedMinimumHeight/getSuggestedMinimumWidth。

对于ViewGroup，需要调用measureChildren以测量孩子们的大小，并根据孩子们的大小来计算自己的大小。
并且，至少要考虑margin的大小，可遍历每个child，并调用measureChild测量它们的大小，再调用child.getLayoutParams();获取LayoutParams。

4. 实现onLayout方法（自定义ViewGroup实现）

自定义View不需要实现该方法，因为View的位置是有包含它的ViewGroup指定的。

自定义ViewGroup在该方法中指定children的位置，通过调用child的layout方法。
自定义ViewGroup的宽和高应该从onLayout方法的参数获取，而不是使用Measure width/height。

5. 实现onDraw方法（自定义View实现）

对于ViewGroup则不需要实现该函数，因为作为容器是“没有内容“的，其包含了多个子view，而子View已经实现了自己的绘制方法。

自定义View的宽和高应该调用getWidth和getHeight获取，而不是使用Measure width/height。
要考虑padding的大小，根据需要是否绘制到Padding的内部。
在onDraw方法中，不要创建对象，这样会消耗大量内存。
